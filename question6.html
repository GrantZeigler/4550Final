1. Factory Method vs. Abstract Factory

  • Problems they solve
    • Factory Method:
      • Developer needs to create objects while allowing for different subclasses to be chosen
    • Abstract Factory:
      • The developer has several factory methods used, and wants to place them together under one interfact\e
  • Solution
    • Factory Method:
      • Create an interface that creates objects
      • The objects are all different subclasses
    • Abstract Factory:
      • Encapsulate groups of factories that have a common theme to create objects that are part of the theme
  • Consequences
    • Factory Method:
      • Code only deals with interfaces, so it can work with any subclass
      • Requires a subclass creator
    • Abstract Factory:
      • Client does not know which objects it is getting, since it only deals with interfaces
      • It is easy to use the interface, since similar factories are grouped together

 

2. Adapter vs Decorator

  • Problem
    • Adapter:
      • Used when incompatible interfaces need to work together
    • Decorator:
      • Used when responsibilities need to be added/removed dynamically
  • Solution
    • Adapter:
      • Certain processes are ran though the adapter, which converts the necessary information to whatever the adaptee wants
    • Decorator:
      • Creates a decorator object that has operations ran through it, and can be changed as needed
  • Consequences
    • Adapter:
      • Class adapter: adapter is a subclass of the adaptee, and can override some of the behavior
      • Object Adapter:  Single adapter can work with a class and its subclasses, harder to override adaptee behavior
    • Decorator:
      • Allows for responsibilities to change at runtime
      • Makes the program more flexible

 

3. Iterator vs Visitor

  • Problem
    • Iterator: 
      • Used when a container needs to be traversed to access the elements
    • Visitor:
      • Used to separate an algorithm from an object, and to add operations without modifying the structure
  • Solution
    • Iterator:
      • Create a general iterator that keeps track of its location and what is there
    • Visitor:
      • Defines a visitor object containing an operation that is performed on other elements of the structure
  • Consequences
    • Iterator:
      • Container specific, cannot be decoupled from the container
      • General iterators can be created to work with subclasses
    • Visitor:
      • Algorithms involving several classes can be managed in a single location
      • New classes need to be compatible with the visitor
